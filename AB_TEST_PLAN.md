# План A/B-тестирования моделей

## Цель теста

Проверить, даёт ли новая модель `v2` более полезные прогнозы дефолта по сравнению с текущей моделью `v1`.

Контрольная группа:

- модель `v1`;
- Logistic Regression.

Тестовая группа:

- модель `v2`;
- Random Forest Classifier.

## Разделение трафика

Для production-сценария предлагается случайное разделение трафика 50/50:

- 50% клиентов отправляются на `v1`;
- 50% клиентов отправляются на `v2`.

Разделение лучше делать стабильно по `client_id`, чтобы один и тот же клиент всегда попадал в одну и ту же группу. Например:

```text
hash(client_id) % 2 == 0 -> v1
hash(client_id) % 2 == 1 -> v2
```

В учебной реализации модель можно выбрать через поле `model_version` в JSON-запросе.

## Продолжительность теста

Минимальная продолжительность теста — 2-4 недели. Этого достаточно, чтобы собрать данные по разным дням недели и избежать перекоса из-за краткосрочных колебаний.

Для реального кредитного дефолта полный результат может быть виден только через месяц, так как таргет связан с дефолтом в следующем месяце. Поэтому финальную бизнес-оценку корректнее делать после накопления фактических исходов.

## Основные метрики

### Основная метрика

`F1-score` для класса дефолта.

Эта метрика выбрана, потому что классы несбалансированы, а для банка важны и полнота нахождения дефолтных клиентов, и точность таких срабатываний.

### Дополнительные метрики

1. `Precision` для класса дефолта.

Precision важен, потому что ложное срабатывание может привести к отказу хорошему клиенту.

2. `Recall` для класса дефолта.

Recall важен, потому что пропущенный дефолтный клиент может привести к финансовым потерям.

## Бизнес-метрики

### 1. Expected Loss

Для каждого клиента можно оценить ожидаемые потери:

```text
Expected Loss = PD * EAD * LGD
```

Модель считается полезнее, если помогает снизить ожидаемые потери при сопоставимой доле одобренных заявок.

### 2. Approval Rate при фиксированном уровне риска

Можно сравнить, какая модель позволяет одобрить больше заявок при одинаковом максимальном уровне ожидаемого дефолта.

## Статистический анализ

Для сравнения долей ошибок и метрик классификации можно использовать z-test для пропорций. Например, можно сравнить долю правильно обнаруженных дефолтов или долю ложных срабатываний между группами.

Для средних бизнес-показателей, например Expected Loss на клиента, можно использовать t-test. Если распределение сильно скошено, лучше дополнительно использовать bootstrap-доверительные интервалы.

Доверительные интервалы предлагается строить на уровне 95%.

## Критерий успешности

Модель `v2` считается успешной, если выполняются условия:

- F1-score по классу дефолта статистически значимо выше, чем у `v1`;
- Precision не ухудшается относительно `v1`;
- бизнес-метрика Expected Loss не хуже контрольной группы;
- технические метрики сервиса не ухудшаются: нет роста ошибок и задержек API.

## Связь с архитектурой

API должен поддерживать две версии модели. В проекте это реализовано через параметр `model_version`.

В production-версии дополнительно нужно логировать:

- id клиента или анонимный идентификатор;
- версию модели;
- вероятность дефолта;
- итоговое решение;
- фактический результат после наступления периода наблюдения.

Без логирования версии модели невозможно корректно сравнить результаты A/B-теста.
