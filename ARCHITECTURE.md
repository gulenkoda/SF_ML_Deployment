# Архитектура сервиса

## Выбор подхода: монолит или микросервисы

Для этого проекта выбран монолитный подход. Причина в том, что сервис решает одну основную задачу: принять JSON с признаками клиента, загрузить модель и вернуть прогноз дефолта. Отдельные микросервисы здесь увеличили бы сложность без большой практической пользы.

Монолит в данном случае проще:

- быстрее запустить локально и в Docker;
- проще отлаживать;
- проще описать в README;
- все компоненты проекта находятся в одном репозитории;
- модель и API разворачиваются вместе.

Микросервисный подход был бы полезен, если бы система стала больше: отдельный сервис авторизации, отдельный сервис фичей, отдельный сервис инференса, отдельный сервис логирования, отдельный сервис мониторинга. Для этого проекта это выглядит избыточно.

## Состав сервиса

Основные части:

- `src/train.py` — обучение моделей и сохранение артефактов;
- `app/model_loader.py` — загрузка моделей из папки `models/`;
- `app/api.py` — Flask API с эндпоинтами `/health` и `/predict`;
- `models/` — сохранённые версии моделей;
- `tests/` — базовые тесты API;
- `Dockerfile` — упаковка сервиса в контейнер.

## A/B-тестирование в архитектуре

В API добавлено поле `model_version`. Благодаря этому можно явно отправить запрос к версии `v1` или `v2`.

В реальной системе распределение трафика можно было бы сделать несколькими способами:

1. на уровне API Gateway;
2. на уровне балансировщика;
3. внутри самого Flask-приложения по user_id или request_id.

В текущей реализации используется простой вариант: клиент передаёт `model_version`. Это удобно для демонстрации и ручной проверки.

## Брокер сообщений

RabbitMQ или другой брокер очередей мог бы пригодиться при масштабировании проекта.

Возможные сценарии:

- асинхронная обработка больших batch-запросов;
- запись событий инференса в отдельную очередь;
- отправка логов и результатов прогнозов в хранилище;
- отделение API от долгих фоновых задач.

Например, API может быстро принять запрос на batch-предсказание, положить задачу в RabbitMQ и сразу вернуть пользователю `task_id`. Отдельный worker заберёт задачу из очереди, выполнит предсказания и сохранит результат.

## Логирование и мониторинг

В текущем проекте запросы логируются в файл:

```text
logs/api_requests.jsonl
```

Каждая строка — отдельный JSON-объект. В логе сохраняются:

- timestamp;
- request_id;
- endpoint;
- status_code;
- ответ API.

В production-среде эти логи можно собирать через Filebeat и отправлять в ELK-стек. Также можно добавить Prometheus + Grafana для мониторинга технических метрик:

- количество запросов;
- latency;
- доля ошибок 4xx/5xx;
- распределение вероятностей дефолта;
- доля предсказаний класса 1.

## uWSGI / Gunicorn + NGINX

Flask development server не рекомендуется использовать как основной сервер в production. Поэтому приложение обычно запускают через WSGI-сервер, например Gunicorn или uWSGI. Он управляет worker-процессами и лучше подходит для нагрузки.

NGINX в такой схеме используется как reverse proxy. Он может принимать внешний трафик, завершать TLS, отдавать статические файлы, ограничивать размер запросов и проксировать запросы к Flask-приложению.

В Dockerfile этого проекта используется Gunicorn, так как он проще для.

## DVC и MLflow

DVC нужен для версионирования данных и больших артефактов, которые неудобно хранить напрямую в Git. Например, можно версионировать исходный датасет, подготовленные выборки и обученные модели.

MLflow нужен для отслеживания экспериментов. В нём можно хранить параметры обучения, метрики, версии моделей и артефакты. Для этого проекта MLflow был бы полезен при сравнении `v1` и `v2`.

## Бизнес-метрики

Кроме ML-метрик, заказчику важны бизнес-показатели.

### 1. Снижение ожидаемых потерь

Можно оценивать ожидаемые потери по клиентам, которым модель присвоила высокий риск:

```text
Expected Loss = PD * EAD * LGD
```

где:

- `PD` — вероятность дефолта из модели;
- `EAD` — сумма задолженности или лимит;
- `LGD` — доля потерь при дефолте.

### 2. Доля одобренных клиентов при фиксированном риске

Банк может выбрать порог вероятности дефолта и смотреть, сколько заявок можно одобрить, не увеличивая фактическую долю дефолтов выше допустимого уровня.

Эта метрика важна, потому что слишком строгая модель может снижать потери, но одновременно уменьшать прибыль из-за отказов хорошим клиентам.
